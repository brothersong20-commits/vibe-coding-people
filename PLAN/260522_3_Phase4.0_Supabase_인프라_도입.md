# Phase 4.0 — Supabase 인프라 도입 (콘텐츠 DB 이관)

## Context

Vibe-Coding-People은 1~3단계에서 정적 콘텐츠(`content/*.ts` + `content/**/*.mdx`)로 운영되고 있다. 멤버 어드민 폼과 Google Auth는 후속(4.1/4.2)에서 추가되지만, **그 전에 데이터 소스를 Supabase로 옮겨야 한다**. 4.0의 목표는 **UI/디자인은 그대로 두고, 콘텐츠가 DB에서 읽혀오도록 인프라 골격을 깔고 데이터를 이관**하는 것이다. 인증/어드민 UI는 4.1/4.2에서 얹는다 (스키마에 `members` 테이블만 미리 박아둠).

## 결정사항 (확정)

- **범위**: Phase 4.0 — Supabase 프로젝트 + 스키마 + 클라이언트 + 데이터 이관까지. 인증/어드민 UI는 후속.
- **MDX 처리**: 본문을 DB 컬럼(`body_md text`)에 저장, 런타임에 `react-markdown` + `remark-gfm` 렌더. 정적 import (`*BodyMap`) 제거.
- **Supabase 프로젝트**: Supabase MCP로 신규 생성 (리전 `ap-northeast-2`, 이름 `vibe-coding-people`).
- **PK 전략**: 모든 콘텐츠 테이블은 `id uuid` PK + `slug text unique`. 라우트엔 slug만 노출. 후속 단계에서 slug 변경/리네임에 강함.
- **시드 방식**: 일회성 TS 스크립트 (`scripts/seed.ts`, service_role 사용, `upsert({ onConflict: 'slug' })`로 idempotent).
- **렌더링**: `react-markdown` + `remark-gfm`. `mdx-components.tsx` 매핑을 `lib/markdown-components.tsx`로 이식.
- **권한 골격**: 콘텐츠 6개 테이블 + `contest_teams`는 anon SELECT 허용, 쓰기는 service_role만. `members.role = 'admin'` 이메일 화이트리스트(`2026copai@gmail.com` 시드 1행). 정교한 RLS는 4.1.
- **캐싱**: 각 페이지에 `export const revalidate = 60`. Cache Components(`use cache`)는 안정성 위해 4.0.x로 미룸.
- **패키지 매니저**: 현 상태인 **npm** 유지 (lock 기준). CLAUDE.md의 pnpm 표기는 정정.

## 실행 순서

### Step 1 — Supabase 프로젝트 생성 (MCP)
- `list_organizations` → org_id 확인
- `get_cost("project")` → `confirm_cost` → `create_project({ name: "vibe-coding-people", region: "ap-northeast-2" })`
- `get_project_url`, `get_publishable_keys` (URL + anon)
- service_role key는 Supabase 대시보드에서 수동 회수 (MCP 노출 안 됨)

### Step 2 — 환경변수 + .gitignore
- `.env.local` (실제 값), `.env.example` (placeholder) 생성:
  - `NEXT_PUBLIC_SUPABASE_URL`
  - `NEXT_PUBLIC_SUPABASE_ANON_KEY`
  - `SUPABASE_SERVICE_ROLE_KEY` (시드 전용, **절대 `NEXT_PUBLIC_` 금지**)
- `.gitignore`에 `.env*.local` 차단 확인. `.env.example`은 추적.
- Vercel Project Settings → Environment Variables에도 동일 3개 등록 (Preview/Production). `SUPABASE_SERVICE_ROLE_KEY`는 Sensitive 플래그.

### Step 3 — SQL 스키마 마이그레이션 (`apply_migration`)
마이그레이션 이름: `0001_initial_content_schema`. 한 트랜잭션으로 적용.

**공통**
- `set_updated_at()` 트리거 함수
- 각 테이블에 `before update` 트리거

**테이블 스키마**

| 테이블 | 핵심 컬럼 |
|---|---|
| `notices` | id uuid pk, slug unique, title, date date, category check(공지/이벤트/회고), summary, sort_order int, created_at, updated_at |
| `schedule_items` | id, slug unique, start_at timestamptz, end_at timestamptz null, title, location, description, status check(예정/진행중/완료), tags text[] |
| `contests` | id, slug unique, title, host, organizer null, start_date date, end_date date, prize, fields text[], eligibility, url, status check(모집중/예정/마감), tags text[], summary null, body_md text |
| `contest_teams` | id uuid, contest_id uuid fk → contests(id) on delete cascade, name, members text[], status check(모집중/확정/출품완료), note null, github_url null, demo_url null, sort_order int |
| `projects` | id, slug unique, title, summary, tags text[], members text[], thumbnail null, github null, demo null, sort_order int, body_md text |
| `insights` | id, slug unique, title, date date, category check(AI 코딩/도구/워크플로우/회고), summary, tags text[], author, read_time int, body_md text |
| `members` | id uuid pk, email text unique, name null, role check(admin/member) default 'member', google_sub text unique null, created_at, updated_at |

**RLS 골격 (4.0 최소)**
- 7개 콘텐츠 테이블: `enable rls` + `select using (true)`. 쓰기 정책 부재 → service_role만.
- `members`: `enable rls` + `select using (auth.uid()::text = google_sub or role = 'admin')`. 4.1에서 정교화.

**시드 1행**: `members(email='2026copai@gmail.com', role='admin')`

### Step 4 — 패키지 설치 + 클라이언트 helper
```
npm install @supabase/supabase-js @supabase/ssr react-markdown remark-gfm
npm install -D tsx
```

생성:
- `lib/supabase/server.ts` — `createServerClient` (`cookies()` 표준 핸들러). 4.0엔 read-only지만 4.1 대비 표준 형태로.
- `lib/supabase/client.ts` — `createBrowserClient` (4.0 미사용, 4.1 대비).
- `lib/supabase/admin.ts` — service_role. **첫 줄에 `import "server-only"` 강제**.

### Step 5 — 타입 생성
- `mcp__plugin_supabase_supabase__generate_typescript_types` → `lib/database.types.ts`.
- 도메인 타입(`Notice`, `Project`, `Insight`, `Contest`, `ContestTeam`, `ScheduleItem`)은 기존 `content/*.ts`의 형태를 유지해 `lib/types.ts`로 이관. UI는 도메인 타입만 봐서 스키마 변경에 격리.

### Step 6 — 데이터 액세스 레이어 교체
`lib/content.ts`를 DB 쿼리 헬퍼로 전면 재작성 (`import "server-only"` + `cache()` 메모이즈):
- `getNotices()`, `getSchedule()`
- `getOrderedContests()`, `getContestBySlug(slug)` (teams 조인)
- `getOrderedProjects()`, `getProjectBySlug(slug)`
- `getOrderedInsights()`, `getInsightBySlug(slug)`

각 함수 안에 row → 도메인 매핑(`mapNoticeRow` 등). 정렬/필터는 SQL 단(`.order('sort_order')`, `.eq('slug', slug)`).

### Step 7 — 페이지 컴포넌트 수정
패턴이 동일한 변경이므로 한 군데에서 검증 후 나머지에 일괄 적용.

## 위험 / 열린 질문

- **빌드 시 DB 의존**: `generateStaticParams`가 빌드 타임에 Supabase 호출. Vercel build 환경에 env 필수. → 환경변수 Vercel 등록을 Step 2에서 마치므로 OK.
- **MDX frontmatter 유무**: 1단계 MDX 파일에 frontmatter가 들어 있는지 시드 전 grep 확인 — 없으면 `gray-matter`는 그대로 통과.
- **`tsx`의 alias 해석**: `@/content/*` 경로가 시드 스크립트에서 안 풀릴 수 있음 → `tsx`는 보통 `tsconfig.json`의 `paths`를 자동 해석하지만 안 되면 `tsconfig-paths` 추가 또는 상대 경로로 우회.
- **anon 키의 빌드 캐시**: `NEXT_PUBLIC_SUPABASE_ANON_KEY`가 클라이언트 번들에 들어감 (정상). service_role은 절대 서버 함수 밖으로 안 나가는지 grep으로 확인.
- **Korean check 제약**: 한국어 enum 값(`공지/이벤트/회고` 등)을 SQL `check` 제약에 그대로 박는 게 가장 단순하지만, 후속에 enum 타입 만드는 게 깔끔할 수도 있음. 4.0은 단순한 text check로.
